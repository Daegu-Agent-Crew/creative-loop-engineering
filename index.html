창의적 루프 엔지니어링 시스템

프롬프트를 짜는 게 아니라 시스템을 짜는 시대.
선택 몇 개로 창의적 결과물이 자동으로 만들어지는 진화형 파이프라인.

"루프 엔지니어링이란, 에이전트에게 프롬프트를 쓰는 사람 자리에서 당신을 빼내는 것이다.
대신 그 일을 해줄 시스템을 당신이 설계한다."
— Addy Osmani

📋 시스템 개요

이 문서는 "주제·화풍·분위기를 선택하면 중간 피드백 게이트를 거치며 점진적으로 완성되는 창의적 산출물 파이프라인"의 전체 설계도입니다. 기반은 Addy Osmani의 Loop Engineering 6가지 부품 + 기존 랄프톤 생태계의 Phase Split 구조를 통합합니다.

SYSTEM ARCHITECTURE — 전체 시스템 구조도

🧩 루프의 6가지 부품

Addy Osmani가 정의한 루프 엔지니어링의 핵심 부품 5개 + 기억장치 1개를 창의적 파이프라인에 맞게 적용합니다.

기억

💡 창의적 파이프라인에의 적용

이 6가지 부품을 창의적 작업(만화, 일러스트, 디자인 등)에 맞게 재조합합니다. 코드가 아니라 콘텐츠를 만들지만, "발견→분류→수정→검증"의 루프 구조는 동일합니다.

🔄 루프 구조 — 한 바퀴의 4단계

하나의 루프는 발견→분류→수정→검증 네 동작으로 압축됩니다. 각 단계 끝에 자동/수동 피드백 게이트가 있습니다.

LOOP CYCLE — 루프 한 바퀴

📊 5-Phase 창의적 파이프라인

루프의 4단계를 실제 창의적 작업에 맞게 5개 Phase로 확장합니다. 각 Phase 끝에 피드백 게이트(✋)가 있어 주인님이 검토할 수 있습니다.

1 DEFINE

📋 기획 & 설계

주인님이 선택한 주제/화풍/분위기를 바탕으로 PRD를 자동 생성합니다.

  • 주인님 입력: 주제, 소제목, 화풍, 분위기, 포맷, 해상도
  • Planner 서브에이전트가 PRD 자동 생성
  • Researcher가 참고 자료·스타일 리서치
  • 캐릭터 설정 시트, 화풍 가이드, 프롬프트 템플릿 생성
  • 출력: config.json + PRD.md
✋
게이트 1: 기획 승인
주인님이 PRD 확인 → 수정/승인
2 SAMPLE

🎨 샘플 생성 & 화풍 확정

대표 장면 3~5종을 샘플로 생성합니다. 화풀이 확정되는 단계입니다.

  • Writer가 프롬프트 생성 → 이미지 API로 샘플 생성
  • 서로 다른 화풍/구도/빛 조건으로 A/B/C 버전 생성
  • 캐릭터 시트 기반으로 주요 캐릭터 샘플
  • 배경, 분위기, 컬러 팔레트 테스트
  • 출력: samples/v1/ (3~5장)
✋
게이트 2: 화풍 & 방향 승인
"A 버전 좋아" / "B에 C의 색감 섞어" / "다시" → 피드백이 Phase 3 입력으로
3 PRODUCE

🏭 전체 컷 생성 (본제작)

승인된 샘플의 스타일로 전체 컷을 생성합니다. 서브에이전트가 병렬로 작업합니다.

  • 전체 컷 목록(auto-generated from PRD)을 배치
  • Writer 서브에이전트: 컷별 프롬프트 생성
  • 이미지 API 병렬 호출 (timeout 관리)
  • 중간 체크포인트: 10컷마다 Evaluator 자동 검토
  • 컷 간 일관성 검사 (캐릭터 얼굴, 색감, 배경)
  • 출력: output/v1/pages/
✋
게이트 3: 중간 검토
진행 상황 확인 + 방향 수정 가능 / "이거 계속 해" / "방향 바꿔"
4 REFINE

🔧 수정 & 진화

피드백을 반영하여 v(N+1)을 생성합니다. 이 단계가 루프의 핵심입니다.

  • 주인님 피드백을 feedback.md에 기록
  • Critic 서브에이전트: 현재 버전 비판적 평가
  • Writer: 피드백 + Critic 분석 → 수정 프롬프트 생성
  • Evaluator: v(N) vs v(N+1) 비교 평가 → 점수 산출
  • 점수가 기준 미달이면 자동으로 Phase 4 반복 (최대 N회)
  • 출력: output/v2/, scores.json
✋
게이트 4: 최종 승인
"이걸로 간다" (DELIVER) / "아직 부족해" (REFINE 루프)
5 DELIVER

📦 최종 산출 & 메타데이터

최종 산출물과 함께 "무엇이 좋았는지"를 기록합니다. 다음 프로젝트의 진화 재료가 됩니다.

  • 최종 이미지/콘텐츠 정리 및 저장
  • 메타데이터 생성: 어떤 선택이 왜 좋았는지
  • 진화 로그: v1→v2→v3 변경 이력 + 이유
  • 평가 요약: Critic/Evaluator 점수 추이
  • SKILL.md 자동 업데이트 (학습된 패턴 반영)
  • GitHub 배포 / 웹 게시
  • 출력: output/final/, evolution.md
✅
완료 — 진화 데이터가 다음 루프의 입력으로
이번 프로젝트에서 학습한 패턴이 SKILL.md에 누적 → 다음 프로젝트의 품질 기준 상향

🔁 피드백 루프 상세

루프의 핵심은 "만드는 AI와 검사하는 AI를 분리"하는 것입니다. Claude가 자기 결과를 후하게 평가하는 경향(self-bias)을 독립 평가자로 보정합니다.

⚠️ self-bias 보정 데이터 (Addy Osmani 하네스)

Claude 자기 채점: 9.04 / 독립 평가: 7.43 → 1.6점 차이. 같은 글을 자기가 쓰면 후하게 봅니다. 그래서 반드시 독립 Evaluator/Critic를 분리합니다.

📈 진화 구조 — 버전이 누적되면서 학습

매 루프마다 생성된 산출물, 피드백, 평가 점수가 상태 파일에 누적됩니다. 프로젝트가 끝나면 이 진화 기록이 SKILL.md에 반영되어 다음 프로젝트의 기준이 됩니다.

Produce 16/28장 수동 생성 후 멈춤 전체 28컷 자동 생성 + 중간 체크포인트

🎯 핵심 차이

기존: 주인님이 매 단계 직접 지시 → "근데 8장만 생성된거니?" "나머지는?" → 멈춤
루프 적용: 한 번 설정하면 전체 28컷 자동 생성 + 중간에만 주인님이 확인 → 완성

⚖️ 루프 엔지니어링 vs 기존 방식

"매번 프롬프트"와 "루프를 설계"의 차이를 구체적으로 비교합니다.

관점 기존 (프롬프트 엔지니어링) 루프 엔지니어링
역할 실행자 (매 줄 지시) 감독관 (시스템 설계)
재사용 프롬프트 복붙 SKILL.md + 상태 파일 (프로젝트 간)
품질 검증 주인님 눈으로만 서브에이전트 독립 평가 + 주인님 최종 승인
피드백 대화 중 구두로 게이트에서 구조화된 피드백 (기록됨)
진화 사람 머릿속에만 evolution.md에 기록 → 다음 프로젝트에 적용
한계 세션 바뀌면 컨텍스트 리셋 상태 파일로 컨텍스트 유지
주의점 AI 의존도 과대평가 인지적 항복 주의 — 검토는 사람 몫

🚀 구현 계획

OpenClaw의 기존 스킬 생태계 + 루프 엔지니어링을 통합하여 구현합니다.

S1 스킬

SKILL.md 작성

Creative Loop SKILL.md에 프로젝트 인수인계 노트를 작성합니다.

  • Phase별 입력/출력 스펙 정의
  • 프롬프트 템플릿 라이브러리
  • 품질 평가 체크리스트
  • 진화 메타데이터 스키마
S2 에이전트

서브에이전트 구성

Writer/Evaluator/Critic/Planner를 서브에이전트로 분리합니다.

  • Writer: 프롬프트 생성 + 이미지 API 호출
  • Evaluator: 일관성/품질 자동 평가 (별도 세션)
  • Critic: 비판적 분석 + 개선 제안 (별도 세션)
  • Planner: Phase 오케스트레이션 + 게이트 관리
S3 상태

상태 파일 시스템

프로젝트 진행 상태를 디스크에 지속적으로 기록합니다.

  • config.json: 주인님 설정
  • progress.md: 현재 Phase + 완료된 컷
  • feedback.md: 피드백 기록
  • evolution.md: 버전별 변경 이력
  • scores.json: Evaluator/Critic 점수 추이
S4 검증

삼체 EP001로 검증

실제 프로젝트에 적용하여 시스템을 검증합니다.

  • EP001 나머지 12컷을 루프로 자동 생성
  • 중간 피드백 게이트 실제 테스트
  • v1→v2 진화 루프 실증
  • 학습 패턴이 SKILL.md에 자동 누적되는지 확인

📝 최종 목표

"주제: 해적, 화풍: 3D 애니, 분위기: 모험" 한 줄이면 → 샘플 확인 → 자동 전체 생성 → 피드백 루프 → 완성 산출물. 주인님은 감독관 역할만 하시면 됩니다.
