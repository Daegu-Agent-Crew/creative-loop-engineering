## 핵심 철학

**같은 비용으로 더 많이, 더 빠르고, 더 정확하게.**
제어가 아니라 효율.

---

## 1. 컨텍스트 최적화 — 필요한 것만 정확하게

**문제**: Issue 본문 통째로 주면 모델이 잡음에서 핵심을 찾느라 시간/토큰 낭비

**개선**:
- Issue 감지 시 → 본문에서 핵심 요구사항만 추출 → 구조화된 태스크로 변환
- 모델에 들어가는 컨텍스트 = 시스템 규칙 + 구조화된 태스크 + 필요한 리소스만
- 안정적인 프리픽스(시스템 규칙)를 앞에 두면 프롬프트 캐시 적중률↑

**적용**:
```
Issue 본문 (raw)  →  태스크 파서  →  구조화된 태스크 (objective + constraints + scope)
                                   → 모델 입력으로만 이걸 사용
```

---

## 2. 단계 분해 — 한 번에 너무 많지 않게

**문제**: 복잡한 요청을 한 모델 호출에 몰아넣으면 품질↓, 실패율↑

**개선**:
- 복잡한 Issue → 자동 단계 분해
- 각 단계는 하나의 명확한 목표, 하나의 검증 기준
- 이전 단계의 산출물이 다음 단계의 입력

**적용**:
```
큰 Issue → Phase Split 자동화
  Phase 1: 분석 & 계획
  Phase 2: 생성
  Phase 3: 검증 & 수리
각 Phase는 독립 실행, 체크포인트 연결
```

---

## 3. 병렬 실행 — 독립 작업 동시에

**문제**: 읽기/검색을 순차 처리하면 대기 시간 누적

**개선**:
- 독립적인 읽기 작업은 동시 실행
- 참조 수집, 스타일 분석, 이전 에피소드 확인 등 병렬 처리

**적용**:
```
Seq: Issue 분석 → [병렬] 이미지 스타일 참조 / 이전 에피소드 / 프롬프트 분석
                    → [결합] 생성 → [병렬] 품질 검증 / 메타데이터 생성
```

---

## 4. 상태 관리 — 세션 끊겨도 이어서

**문제**: 중간에 세션 끊기면 처음부터 다시 시작 = 비용 낭비

**개선**:
- 각 단계의 산출물을 프롬프트 밖에 저장
- 재시작 시 저장된 상태에서 이어서
- 긴 세션은 compaction으로 컨텍스트 압축, 핵심 상태 유지

**적용**:
```
Issue별 state 파일:
  { phase, checkpoint, artifacts[], errors[], next_step }
세션 재시작 → state 파일 로드 → 마지막 체크포인트에서 재개
```

---

## 5. 스마트 재시도 — 실패해도 똑똑하게

**문제**: 실패하면 그냥 멈추거나 같은 방식으로 재시도

**개선**:
- 일시적 오류(API 타임아웃 등) → 자동 재시도 (최대 2회)
- 근본적 오류(프롬프트 오류, 권한 없음) → 즉시 진단 + 에이전트에 알림
- 모델이 잘못된 툴 콜 → 에러 관찰을 돌려주고 자기 수정 유도

**적용**:
```
실패 분류:
  transient (타임아웃, 레이트리밋) → 3초 후 재시도, 최대 2회
  permanent (권한없음, 잘못된인자) → 에러 관찰 반환, 모델이 스스로 교정
  budget   (비용초과, 단계초과)   → 안전 중지 + 진행 상태 저장
```

## 📊 기대 효과

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 컨텍스트 비용 | Issue 통째로 주입 | 구조화 태스크만 → 40~60% 감소 |
| 복잡한 Issue 성공률 | 한 번에 몰아넣기 | 단계 분해 → 2~3배 향상 |
| 처리 속도 | 순차 처리 | 병렬 읽기 → 2~3배 |
| 실패 복구 | 처음부터 재시작 | 상태 복원 → 80% 비용 절감 |
| 재시도 효율 | 같은 방식 재시도 | 에러 분류별 정책 → 실패율 ↓ |
