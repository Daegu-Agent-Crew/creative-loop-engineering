## 1. 프로젝트 개요

### 1.1 배경
Addy Osmani의 Loop Engineering 6가지 부품 + 기존 랄프톤 생태계 Phase Split 구조를 통합한 **창의적 루프 엔지니어링 시스템**의 다중 사용자용 UI를 설계합니다.

### 1.2 스코프 정의 (v1.1 추가)

| 단계 | 정의 | 기술 |
|------|------|------|
| **Phase 1: 프로토타입** | 정적 데모, 개인 localStorage 기반, GitHub Pages 배포 | Vanilla SPA |
| **Phase 2: 커뮤니티** | GitHub 로그인, 갤러리/템플릿 공유, Issues 기반 저장 | GitHub API |
| **Phase 3: 제품** | 서버 API, 작업 큐, 실시간 알림, 팀 협업 | Cloudflare Workers + DB |

> 본 PRD는 **Phase 1~2**를 중심으로 작성. Phase 3은 별도 PRD로 분리.

### 1.3 목표
- **"프롬프트를 잘 쓰는 도구가 아니라, 좋은 결과를 고르는 도구"** (v1.1 차별화 메시지)
- 비기술자도 3단계 카드 선택으로 창의적 산출물 생성
- 반복 개선 루프: 후보 비교 → 선택 → 피드백 반영 → 품질 상승

### 1.4 대상 사용자
- 1차: 대구 에이전트 빌더 커뮤니티 멤버 (10+명, GitHub 계정 보유)
- 2차: 일반 창작자 (만화, 일러스트, 글쓰기)
- 3차: 팀 기반 콘텐츠 스튜디오 (Phase 3)

### 1.5 차별화 포인트 (v1.1 추가)

**"한 장 생성보다, 시리즈 품질 일관성에 강한 창작 루프"**

| 경쟁사 | 약점 → 우리의 강점 |
|--------|-------------------|
| Midjourney | 개인 생성 중심, 시리즈 일관성 약함 → **루프 기반 반복 개선으로 스타일 유지** |
| Canva AI | 템플릿 편집 위주, 생성 파이프라인 없음 → **5-Phase 구조화된 생성 워크플로우** |
| Leonardo/Krea | 프롬프트 의존적 → **카드 선택 UI로 비전문가도 고품질 결과** |

## 2. 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **선택 우선** | 텍스트 입력 최소화, 카드/슬라이더/버튼으로 선택 |

| **모바일 동등** | 카드 선택 + 승인만 되면 모바일에서도 충분 |

## 4. 상세 스펙

### 4.1 랜딩 페이지 (`/`)

**목표:** 30초 안에 "이게 뭔지 + 해보고 싶다" 느끼게

- Hero: 애니메이션 루프 비주얼 (Phase들이 순환하는 SVG 애니메이션)
- 차별화 헤드라인 (v1.1): *"프롬프트를 쓰지 마세요. 고르세요."*
- "바로 체험하기" CTA → 가입 없이 바로 /new로 이동
- 갤러리에서 최근 완성 작품 3~4개 썸네일
- 카테고리 태그: 만화 / 일러스트 / 스토리보드 / 글쓰기

**핵심: 3단계 카드 선택 UI**

│  └─────────┘ └─────────┘ └─────────┘        │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │

│  └─────────┘ └─────────┘ └─────────┘        │
│                                              │
│          선택 시 오른쪽에 실시간 프리뷰         │

│                                              │

- **카드 선택:** 클릭 한 번, 다중 선택 가능 (예: SF + 역사 혼합)

- **실시간 프리뷰:** 사전 샘플 이미지 (v1.1 명확화 — 즉석 생성 아님, 비용 없음)
- **"AI가 추천해줘" 버튼:** 랜덤/트렌딩 조합 자동 선택
- **예상 시간/결과물 수 안내** (v1.1 추가): 기대치 조정

### 4.5 템플릿 마켓 (`/templates`)

- 완성된 프로젝트의 설정값(주제+화풍+분위기+피드백 칩)을 템플릿으로 저장

- **복제 후 수정 흐름** (v1.1 명확화): 3단계 카드가 미리 선택된 상태로 /new 진입, 각 단계별로 덮어쓰기 가능
- 카테고리별 정렬, 다운로드 수 랭킹
- GitHub Discussions 기반 (Phase 2)

### 4.6 게스트 세션 복구 (v1.1 추가)

**문제:** 가입 없이 체험 중 브라우저를 닫으면 프로젝트가 유실됨

**해결:**
- 프로젝트 생성 시 `/resume/:token` 형태의 복구 URL 자동 발급
- URL을 localStorage에 저장 + 클립보드 복사 안내 표시
- 복구 URL 접근 시 sessionStorage에서 프로젝트 상태 복원
- **주의:** localStorage 용량 한계(~5MB) → 이미지는 base64가 아닌 참조 URL만 저장
- GitHub 계정 연동 시 기존 게스트 프로젝트를 계정으로 마이그레이션 제안

## 5. 사용자 플로우

### 5.1 신규 사용자 (First Experience) — v1.1 개정

```
랜딩 → "체험하기" 클릭
  → 주제 카드 선택 (1클릭)
  → 화풍 카드 선택 (1클릭)
  → 분위기 카드 선택 (1클릭)
  → 예상 시간/결과물 수 확인 ← (v1.1)
  → "생성 시작" 클릭
  → 진행 대시보드: 상태 표시 (대기/생성/실패) ← (v1.1)
  → 후보 4개 중 선택
  → 구조화 피드백 칩 선택 + 별점 ← (v1.1)
  → [승인] → 다음 Phase 자동 진행
  → 완성!
  → 공개 여부 선택 (기본: 비공개) ← (v1.1)
  → 갤러리 공개 선택 시 → 게시
  → 복구 URL 안내 + "계정 연결하면 나중에도 보존" 제안 ← (v1.1)
```

**목표 시간:** 첫 결과물까지 5분 이내

### 5.2 숙련 사용자

```
대시보드 → 진행 중 프로젝트 확인
  → 게이트에서 구조화 피드백 칩 빠른 선택 ← (v1.1)
  → 여러 프로젝트 병렬 진행
  → 템플릿으로 빠른 시작 (복제 → 카드 덮어쓰기) ← (v1.1)
  → 갤러리에서 영감 수집
```

## 6. 기술 아키텍처

### 6.1 프론트엔드

| 항목 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Vanilla HTML/CSS/JS (SPA) | GitHub Pages 배포, 의존성 최소 |
| 스타일 | CSS Variables + Grid/Flex | 기존 index.html과 일관 |
| 상태관리 (Phase 1) | localStorage + sessionStorage | 프로토타입용 |
| 상태관리 (Phase 2) | GitHub API + local cache | 다중 기기 동기화 |
| 라우팅 | hash-based SPA router | GitHub Pages 호환 |

### 6.2 GitHub 활용 전략 (v1.1 구체화)

| 기능 | GitHub 기능 | Phase |
|------|------------|-------|
| 프로젝트 저장 | Issues (JSON body) | Phase 1: local / Phase 2: Issues |
| 갤러리 | Issues + labels (gallery, category) | Phase 2 |
| 템플릿 | Discussions (카테고리별) | Phase 2 |
| 좋아요 | Issue reactions (👍) | Phase 2 |
| 댓글/피드백 | Issue comments | Phase 2 |
| 사용자 | GitHub OAuth | Phase 2 |
| 이미지 | GitHub Releases assets | Phase 2 (1GB 제약 인지) |
| CI/CD | GitHub Actions (정적 빌드) | Phase 1 |

### 6.3 배포
